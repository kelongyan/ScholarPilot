# ScholarPilot 项目规则

---

## 1. 项目协作原则

ScholarPilot 的开发必须遵循以下原则：

- 可落地优先：先完成稳定闭环，再扩展高级能力。
- 证据优先：任何 AI 生成结果必须尽量保留来源、引用和检索过程。
- 低耦合优先：模块之间通过清晰接口协作，禁止互相穿透内部实现。
- 小步迭代：每次开发只解决明确问题，不顺手扩大范围。
- 可验证交付：没有测试或验证记录，不视为完成。
- 可追踪推进：每个阶段必须有状态、记录、提交和 GitHub 推送。

---

## 2. 阶段开发规则

### 2.1 阶段状态

所有阶段使用统一状态：

- `Not Started`
- `In Progress`
- `Blocked`
- `Review`
- `Done`

### 2.2 阶段开始要求

开始任何阶段前，必须完成：

- 阅读 `doc/分阶段开发方案.md`。
- 明确本阶段目标和不做事项。
- 将阶段状态更新为 `In Progress`。
- 拆分任务清单。
- 明确验证方式。

### 2.3 阶段完成要求

阶段完成必须同时满足：

- 功能完成。
- 测试通过。
- 本地关键流程验证通过。
- 阶段记录已更新。
- 相关文档与代码一致。
- Git commit 已创建。
- 代码已推送到 GitHub。
- 阶段状态标记为 `Done`。

### 2.4 阶段记录

每完成一个阶段，必须在阶段记录文件中标记。建议文件：

```text
doc/开发进度记录.md
```

记录内容必须包括：

- 阶段名称
- 阶段状态
- 完成日期
- 完成功能
- 测试命令
- 测试结果
- 已知问题
- commit hash
- GitHub 分支

## 3. Git 与 GitHub 规则

### 3.1 基本规则

- 修改前先查看 `git status`。
- 只提交与当前任务相关的文件。
- 禁止使用 `git add .` 盲目提交。
- 禁止提交密钥、token、私有配置和大体积无关文件。
- 未经明确确认，不做破坏性 Git 操作。

### 3.2 提交信息

提交信息使用以下前缀：

- `feat:` 新功能
- `fix:` 修复问题
- `docs:` 文档修改
- `refactor:` 重构
- `test:` 测试
- `chore:` 工程配置

示例：

```powershell
git add RULE.md doc/分阶段开发方案.md
git commit -m "docs: add development phases and project rules"
```

### 3.3 推送规则

- 每完成一个阶段，必须推送到 GitHub。
- 默认推送当前主分支，除非项目负责人明确要求使用其他分支。
- 第一次推送需要设置远程跟踪。

示例：

```powershell
git push -u origin main
```

后续推送：

```powershell
git push
```

### 3.4 敏感文件

以下文件禁止提交：

- `.env`
- `.env.local`
- `.env.*.local`
- `credentials.json`
- `secrets.json`
- 私钥文件
- 包含 API key 的配置文件

必须提供示例配置时，使用：

```text
.env.example
```

## 4. 代码结构规则

### 4.1 目录边界

建议项目结构：

```text
backend/
frontend/
doc/
RULE.md
ai-research-copilot-proposal.md
```

后端内部建议：

```text
backend/app/
├─ api/
├─ core/
├─ models/
├─ schemas/
├─ services/
├─ repositories/
└─ main.py
```

### 4.2 分层职责

| 层 | 职责 |
|---|---|
| `api` | HTTP 路由、请求参数、响应格式 |
| `schemas` | Pydantic DTO、输入输出结构 |
| `services` | 业务流程编排 |
| `repositories` | 数据访问 |
| `models` | 数据库模型 |
| `core` | 配置、日志、异常、通用基础设施 |

### 4.3 禁止事项

- API 层直接操作数据库。
- API 层直接拼接复杂业务逻辑。
- Service 层直接依赖具体 Web 框架对象。
- Repository 层调用 LLM 或外部 API。
- 前端组件直接硬编码后端地址。
- 在多个模块复制同一段复杂逻辑。

---

## 5. 低耦合开发规范

### 5.1 单一职责

每个模块只做一类事情：

- Parser 只负责解析文档。
- Chunker 只负责切分文本。
- Embedding Service 只负责向量生成。
- Retriever 只负责检索。
- Reranker 只负责重排。
- Chat Service 只负责编排证据和生成回答。

### 5.2 接口隔离

外部能力必须通过接口封装：

- LLM provider
- embedding provider
- vector store
- document parser
- external paper search provider

业务代码不能直接绑定某一个具体模型、数据库或第三方服务。

### 5.3 依赖方向

依赖方向必须稳定：

```text
api -> services -> repositories
services -> providers
services -> domain models
```

禁止反向依赖：

- repository 不依赖 service。
- provider 不依赖 api。
- core 不依赖业务模块。

### 5.4 可替换性

以下组件必须预留替换空间：

- FAISS / Qdrant / Milvus
- OpenAI / Qwen / DeepSeek / local LLM
- PyMuPDF / Docling / GROBID
- BM25 / Elasticsearch / sparse vector

## 6. RAG 开发规范

### 6.1 证据链要求

RAG 输出必须尽量包含：

- `doc_id`
- `chunk_id`
- 文档标题
- 章节
- 页码
- 原文片段
- 检索分数

### 6.2 禁止无证据生成

当检索证据不足时，系统必须明确说明：

```text
当前资料不足以支持可靠回答。
```

禁止让模型编造论文结论、实验结果、指标数值或引用来源。

### 6.3 Chunk 规则

chunk 必须保留来源信息：

- 文档 ID
- 页码范围
- 章节名
- chunk 序号
- chunk 类型

### 6.4 检索 trace

重要问答必须记录：

- 原始 query
- 改写 query
- dense retrieval 结果
- sparse retrieval 结果
- rerank 结果
- 最终 evidence pack

---

## 7. Agent 开发规范

### 7.1 受控 Agent

Agent 必须运行在受控工作流中，不允许无限自主循环。

每个 Agent 必须有：

- 明确输入
- 明确输出
- 明确工具权限
- 最大迭代次数
- 失败返回方式

### 7.2 Agent 职责

- Planner：任务拆解，不生成最终答案。
- Retriever：检索和工具调用，不做结论编造。
- Evidence Synthesizer：归纳证据，不创建不存在的证据。
- Reviewer：检查引用和不确定性，不擅自扩大结论。

### 7.3 工具调用

- 工具调用必须记录 trace。
- 高成本或外部请求较多的任务必须限流。
- 涉及删除、覆盖、批量写入的动作必须人工确认。

## 8. 测试与验证规则

### 8.1 后端测试

后端至少包含：

- 单元测试
- API 集成测试
- RAG 检索测试
- 文档解析测试

常用命令示例：

```powershell
pytest
```

### 8.2 前端验证

前端至少包含：

- 类型检查
- lint
- 关键页面手动验证

常用命令示例：

```powershell
npm run lint
npm run build
```

### 8.3 阶段验收

每个阶段必须记录实际运行过的命令和结果。不能只写“已测试”。

### 8.4 无法测试时

如果因为依赖、账号、模型额度或外部服务不可用导致无法测试，必须记录：

- 未测试内容
- 原因
- 风险
- 后续补测方式

## 9. 文档规则

### 9.1 文档创建原则

默认不随意新增文档。只有以下情况允许新增：

- 用户明确要求。
- 阶段开发必须记录。
- 接口、部署、配置或架构发生实质变化。

### 9.2 文档更新要求

代码行为变化后，如果影响以下内容，必须同步更新文档：

- 启动方式
- 环境变量
- API 路径
- 数据模型
- 阶段状态
- 验收方式

### 9.3 禁用空泛文档

文档必须写清楚实际决策和操作方法，禁止只写口号。

---

## 10. 配置与安全规则

### 10.1 环境变量

密钥和环境差异配置必须通过环境变量管理。

禁止在代码中硬编码：

- API key
- token
- 数据库密码
- 私有路径
- 云服务凭证

### 10.2 示例配置

需要说明配置项时，使用 `.env.example`：

```text
OPENAI_API_KEY=
DATABASE_URL=
REDIS_URL=
VECTOR_STORE_URL=
```

### 10.3 外部文档安全

上传文档、网页内容、README 内容都视为不可信输入。

系统必须防范：

- prompt injection
- 恶意链接
- 伪造指令
- 超长文本攻击
- 异常文件格式

---

## 11. 前端开发规范

### 11.1 UI 原则

- 优先清晰和高效，不做过度装饰。
- 三栏布局围绕文档、对话、引用证据展开。
- 用户必须能看到答案依据。
- 长任务必须展示状态。
- 错误信息必须可理解。

### 11.2 组件规则

- 组件职责单一。
- 数据请求放到统一 client 或 hook 中。
- 不在组件里硬编码复杂业务规则。
- 表单必须有基础校验。
- 加载、空状态、错误状态必须处理。

---

## 12. 后端开发规范

### 12.1 API 规则

- 路由命名清晰。
- 响应结构稳定。
- 错误返回统一格式。
- 请求参数使用 Pydantic schema。
- 不在路由函数中写复杂业务逻辑。

### 12.2 Service 规则

- 一个 service 只负责编排一个业务领域。
- service 依赖接口，不直接依赖具体第三方实现。
- 复杂流程拆成明确步骤。
- 失败原因必须可追踪。

### 12.3 Repository 规则

- 只处理数据读写。
- 不调用模型。
- 不调用外部搜索 API。
- 不包含业务决策。

---

## 13. 性能与成本规则

### 13.1 成本控制

- embedding 结果应缓存。
- 相同文档不重复索引。
- 长任务要限制最大 token 和最大检索轮数。
- 外部 API 请求要有重试和限流。

### 13.2 性能目标

MVP 阶段建议目标：

- 单篇论文索引可在可接受时间内完成。
- 普通问答响应尽量控制在数秒级。
- 长任务必须异步执行。

---

## 14. 代码质量要求

### 14.1 必须做到

- 命名清晰。
- 函数长度适中。
- 模块职责明确。
- 删除无用代码。
- 删除无用导入。
- 错误处理可读。
- 日志包含必要上下文。

### 14.2 不允许

- 为一次性逻辑过度抽象。
- 未经需要引入新依赖。
- 大量复制粘贴。
- 用全局变量隐藏状态。
- 捕获异常后静默忽略。
- 把临时调试代码提交。

---

## 15. 开发完成检查清单

每次开发完成前，检查：

- [ ] 是否只改了当前任务需要的文件。
- [ ] 是否有无用导入、无用变量、无用函数。
- [ ] 是否保持模块低耦合。
- [ ] 是否更新必要文档。
- [ ] 是否运行测试或说明无法测试原因。
- [ ] 是否查看 `git status`。
- [ ] 是否只 stage 相关文件。
- [ ] 是否创建清晰 commit。
- [ ] 阶段完成时是否推送 GitHub。

---

## 16. 当前最高优先级

现阶段最高优先级是完成 Phase 0 和 Phase 1：

```text
项目骨架 -> PDF 上传 -> 解析 -> chunk -> embedding -> 检索 -> 问答 -> 引用返回
```

在该闭环稳定前，不优先开发复杂 GraphRAG、完整多 Agent、自动长篇写作和大规模部署能力。
