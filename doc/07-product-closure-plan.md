# Kairos 产品闭环分阶段开发方案

> 本文件基于当前代码实现和现有路线图，拆解从“已可用”到“可运营、可评测、可部署”的后续推进方案。它补充而不是替代 [doc/02-development-roadmap.md](doc/02-development-roadmap.md)。

---

## 1. 方案定位

Kairos 现在已经具备核心 RAG、知识库层、反馈、审计、评测、受控 Agent 的基础能力，但还没有完全形成一个“问题发现 - 修复 - 复测 - 观察 - 迭代”的完整产品闭环。

这份方案的目标不是继续扩大概念，而是把当前能力收拢成一个能持续运营的知识平台。

最终要达成的状态是：

- 用户能在知识库范围内完成上传、检索、提问、反馈、修复和复测。
- 系统能把坏答案、坏引用、失败文档、失败 Agent 运行自动转成可处理项。
- 每一次答案、评测、Agent 运行和关键操作都能追踪、审计和比较。
- 文档生命周期、权限、版本、部署和监控都能支撑真实团队使用。

---

## 2. 当前基线

结合代码和现有文档，当前状态可以概括为：

- 核心 RAG 链路已通：query rewrite、dense retrieval、BM25、RRF、rerank、Evidence Pack、grounded answer。
- 知识库层已通：KB 创建、文档归属、KB 范围问答、KB 过滤。
- 反馈和知识运营已起步：question log、feedback、knowledge operation item、Agent review item。
- 观测和审计已起步：chat trace、Agent run、audit log、evaluation run。
- 受控 Agent 已起步：固定步骤、步骤 trace、运行历史、过滤和复看。
- 权限已起步：最小 RBAC、KB scope、前端角色门控。

当前不足主要集中在：

- 运营项还没有变成真正的修复动作闭环。
- 评测还偏回归，不足以衡量真实质量趋势。
- 权限还不是生产级组织系统。
- 文档仍然是 PDF 优先，生命周期管理不完整。
- 观测数据还没有形成管理视角的看板。
- Agent 还是受控工作流切片，不是成熟编排层。

## 3. 阶段总览

| 阶段 | 目标 | 完成后的能力 |
|---|---|---|
| 阶段 0 | 基线冻结 | 现有能力不回退，接口和测试边界稳定 |
| 阶段 1 | 知识运营闭环 | 坏答案、坏引用、失败文档都能进入处理流 |
| 阶段 2 | 评测与可观测闭环 | 每次问答和 Agent 运行都能复盘和对比 |
| 阶段 3 | 权限与治理闭环 | 访问控制、成员关系、审计覆盖真实可用 |
| 阶段 4 | 文档生命周期闭环 | 多格式接入、版本、替换、删除、重建可控 |
| 阶段 5 | 检索与回答质量提升 | 证据更准、引用更稳、拒答更明确 |
| 阶段 6 | 受控 Agent 稳定化 | 复杂任务可分步执行，但始终受控 |
| 阶段 7 | 生产化与持续优化 | 可部署、可监控、可恢复、可持续迭代 |

推荐顺序：`0 -> 1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7`

---

## 4. 分阶段方案

### 阶段 0：基线冻结

- 目标：锁定当前可用能力，避免后续扩展破坏现有 RAG 和 Agent 契约。
- 主要任务：固定 API 入参/出参、Pydantic schema、前端交互路径和核心回归测试。
- 交付物：接口契约说明、回归测试清单、已知问题清单、发布检查表。
- 验收标准：上传、问答、反馈、运营项、评测、Agent 历史、审计这些主路径全部通过回归。

### 阶段 1：知识运营闭环

- 目标：把“问题信号”真正变成“可处理工单”。
- 主要任务：完善 knowledge operation item 的状态机和类型，打通问题日志、负反馈、失败文档、Agent 预警到运营项的自动生成。
- 主要任务：把处理动作和系统动作连起来，例如重新索引、补文档、生成 FAQ 草稿、修正引用、忽略误报。
- 主要任务：补齐运营项的去重、聚合、处理备注、处理历史和与来源对象的关联。
- 交付物：运营项处理台、处理状态流转、处理结果留痕、可追踪的运营日志。
- 验收标准：每个无答案问题、差反馈、失败文档和异常 Agent 运行都能生成可处理项，并且可以闭环关闭。

### 阶段 2：评测与可观测闭环

- 目标：让质量不是“感觉”，而是可量化、可对比、可回归。
- 主要任务：把 chat trace、Agent trace、evaluation run、question log 的关联做完整，尽量避免 best-effort 丢失。
- 主要任务：补充评测维度，至少覆盖命中率、Recall@K、MRR、Citation Accuracy、Faithfulness、Answer Relevance、延迟和成本。
- 主要任务：让评测结果可比较，记录 dataset version、prompt version、retriever config version 和运行环境摘要。
- 主要任务：做管理视角看板，至少展示质量趋势、运营 backlog、无答案率、低质量引用、延迟分布、评测回归。
- 交付物：可复跑评测集、评测报告、趋势看板、运行历史对比页。
- 验收标准：同一数据集可重复运行，结果可比较，问题可定位，质量回退可被看出来。

### 阶段 3：权限与治理闭环

- 目标：把现在的最小 RBAC 升级成真正可上线的组织治理能力。
- 主要任务：补齐用户、角色、知识库成员关系、owner、visibility、邀请和撤权。
- 主要任务：把文档、知识库、反馈、运营项、评测、审计和 Agent 运行的写路径都纳入权限和审计。
- 主要任务：让前端和后端一致地做权限收敛，避免只靠 UI 隐藏按钮。
- 主要任务：明确知识库范围的访问政策，避免跨 KB 数据串读。
- 交付物：真实登录/鉴权方案、成员管理能力、权限矩阵、审计覆盖表。
- 验收标准：未授权用户不能访问受限知识库和管理功能，关键写操作都能审计到。

### 阶段 4：文档生命周期闭环

- 目标：从“PDF 上传”升级到“知识资产治理”。
- 主要任务：扩展多格式接入，优先补 DOCX、Markdown、HTML、TXT、PPTX，复杂 PDF 再补 OCR 路径。
- 主要任务：补齐文档版本、替换、删除、归档、恢复和重建索引能力，确保旧向量和旧 chunk 不残留。
- 主要任务：把不同格式的解析、切块和元数据提取做成可替换的 parser/provider 结构。
- 主要任务：处理重复文档、重复片段和过期文档的清理策略。
- 交付物：多格式导入、文档版本管理、替换/删除流程、索引清理机制。
- 验收标准：任意文档变更后，知识库检索结果不会继续命中旧数据，文档生命周期可追踪。

### 阶段 5：检索与回答质量提升

- 目标：让答案更准、引用更稳、拒答更明确。
- 主要任务：引入真实 reranker provider，替换或增强当前 deterministic fallback。
- 主要任务：提升 query rewrite、检索融合和候选排序质量，必要时做查询意图分流。
- 主要任务：把多文档知识库问答做稳，避免仍然只偏向单文档思路。
- 主要任务：增加引用校验和证据覆盖检查，避免“看起来像引用”的错误答案。
- 主要任务：强化 insufficient evidence 规则，证据不足时明确拒答，不硬编。
- 交付物：质量提升版本、检索对比报告、引用校验结果、拒答策略说明。
- 验收标准：固定评测集上的质量指标稳定提升，引用与证据一致，低证据场景能稳定拒答。

### 阶段 6：受控 Agent 稳定化

- 目标：把 Agent 保持在“能帮助做复杂任务，但不会失控”的边界内。
- 主要任务：继续保留固定工作流和最大步数限制，保证每一步都有明确输入、输出和状态。
- 主要任务：完善 Planner、Retrieval、Analyst、Writer、Reviewer 的步骤契约和失败回退。
- 主要任务：把 reviewer 的风险信号真正反映到运营项和审计里。
- 主要任务：只有在当前边界稳定后，才评估是否用 LangGraph 替换内部编排实现。
- 交付物：稳定的 Agent 工作流、步级 trace、失败回退策略、Agent 运营项。
- 验收标准：简单问题走短链路，复杂问题走受控多步链路，失败时能安全退化，最终仍保留引用。

### 阶段 7：生产化与持续优化

- 目标：让系统进入可部署、可监控、可恢复、可运营的状态。
- 主要任务：补齐容器化部署、环境配置、密钥管理、健康检查、限流、备份和恢复演练。
- 主要任务：把监控从“能看”升级为“能告警、能定位、能回溯”。
- 主要任务：补充使用量、质量趋势、知识库增长、运营 backlog、评测回归和成本统计看板。
- 主要任务：整理发布流程和运行手册，形成长期维护机制。
- 交付物：部署说明、运行手册、监控看板、备份恢复方案、发布检查表。
- 验收标准：系统可一键启动，关键服务健康可见，数据可备份恢复，问题可被快速定位。

---

## 5. 最终闭环标准

当下面这些条件都满足时，Kairos 可以认为完成了完整产品闭环：

- 用户能在知识库范围内完成导入、提问、反馈、修复、复测。
- 系统能把低质量信号自动沉淀成运营项，并支持处理和关闭。
- 每一次问答、评测和 Agent 运行都有 trace、审计和可比结果。
- 文档生命周期、权限、版本和删除清理都不会留下脏数据。
- 质量趋势、运营 backlog、知识缺口和成本变化都能在看板里看见。
- 系统可以被稳定部署、监控、备份和恢复。

---

## 6. 控制边界

为了避免范围再次膨胀，以下方向建议继续后置：

- 不急于引入 GraphRAG、A2A、MCP 或开放式多 Agent 市场。
- 不急于把知识平台重做成通用聊天平台。
- 不急于替换当前稳定的 FastAPI、Next.js、PostgreSQL、Qdrant、Redis/RQ 主栈。
- 不急于接入额外的大型基础设施，除非当前链路已经出现明确瓶颈。
